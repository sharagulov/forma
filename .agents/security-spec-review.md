# Security spec review: FORMA Architects

**Входной документ:** `.agents/architecture.md`  
**Этап:** спецификация (код приложения не реализован)

---

## Краткое резюме

Архитектура описывает статический портфолио-сайт на Next.js 15 без обязательного backend и БД. Основные зоны риска на уровне спецификации: **опциональный** `POST /api/contact` (спам, злоупотребления, CSRF, инъекции в email), **экспозиция публичных токенов** (Mapbox), **отсутствие явных требований** к заголовкам безопасности (CSP, HSTS), **передача данных третьим лицам** при опции со сторонними формами, **логирование** содержимого заявок. Для ветки без серверного API поверхность атаки минимальна; при выборе опции B необходимо заранее зафиксировать контроли в ТЗ/архитектуре.

## Находки

| ID | Категория (OWASP) | Описание | Severity | Рекомендация | Раздел architecture.md |
|----|-------------------|----------|----------|--------------|-------------------------|
| SEC-001 | A05 Security Misconfiguration / A02 Cryptographic Failures | Публичный `NEXT_PUBLIC_MAPBOX_TOKEN` попадает в клиентский бандл по замыслу; при утечке или злоупотреблении возможны расходы квоты и несанкционированное использование ключа. | MEDIUM | В Mapbox задать **URL restrictions** (allowed URLs) и ограничения по scope; не использовать секретные (server) ключи на клиенте; план ротации токена при компрометации. | §2.4, §5 |
| SEC-002 | A04 Insecure Design / A07 Identification and Authentication Failures | Для опции B (`POST /api/contact`) в спецификации нет требований к **rate limiting**, защите от спама и автоматизированного злоупотребления. Публичный endpoint без лимитов приводит к DoS провайдера почты, исчерпанию квот и флуду получателя. | HIGH | Зафиксировать в реализации: лимит запросов по IP/сессии (edge/middleware или хранилище счётчиков), при необходимости honeypot или CAPTCHA/hCaptcha; мониторинг ошибок провайдера. | §2.2 |
| SEC-003 | A01 Broken Access Control (CSRF) | Серверная отправка формы через Same-Origin fetch из браузера создаёт классический **CSRF**-риск для state-changing POST без дополнительных мер в спецификации. | HIGH | Для опции B: **SameSite=Lax/Strict** для cookie-сессий (если появятся), либо **CSRF-токен** (double-submit или синхронizer), либо проверка `Origin`/`Referer` на сервере с явной политикой отказа; для API-only из браузера — согласовать модель. | §2.2 |
| SEC-004 | A03 Injection | Контактные поля (`name`, `email`, `message`) при передаче в SMTP/API почты могут привести к **email header injection** (переносы строк в заголовках) и смежным проблемам, если сервер не нормализует ввод. | HIGH | Серверная валидация: запрет/экранирование `\r\n` в полях, попадающих в заголовки; жёсткие лимиты длины; одна «чистая» схема кодирования для провайдера (Resend/SendGrid и т.д.). | §2.2 |
| SEC-005 | A04 Insecure Design / A08 Software and Data Integrity Failures | Опция A полагается на **валидацию только на клиенте** (`mailto:` или внешний сервис); для `mailto:` злоумышленник не бьёт по серверу проекта, но пользовательский опыт и целостность данных не гарантируются сервером. | MEDIUM | Если когда-либо появится серверная отправка — дублировать валидацию на сервере; для чистого mailto явно принять ограничение или перейти на B/C. | §2.1 |
| SEC-006 | A05 Security Misconfiguration | В архитектуре не зафиксированы **HTTP security headers** (CSP, HSTS, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, при необходимости `frame-ancestors`). Для Next.js на Vercel часть решается платформой, но CSP для Mapbox/GSAP/шрифтов нужно спроектировать. | MEDIUM | Добавить в реализацию `next.config` headers или middleware: минимально строгий CSP с nonce/hash для скриптов где возможно; HSTS на продакшене; уточнить политику для встраивания карт. | — (общее по стеку §1) |
| SEC-007 | A09 Security Logging and Monitoring Failures | Успешные/неуспешные отправки контактной формы могут привести к **логированию PII** (email, текст сообщения) в Vercel/провайдере логов при отсутствии политики. | MEDIUM | Не логировать тело сообщения и email в открытом виде в общих логах; при отладке — маскирование; ретеншн логов согласовать с политикой конфиденциальности. | §2.2 |
| SEC-008 | A01 Broken Access Control / A05 Security Misconfiguration | Публичный Route Handler без модели «кто может вызывать» — по замыслу открыт; риск **злоупотребления** остаётся (см. SEC-002, SEC-003). | INFO | Явно задокументировать threat model: анонимная отправка допустима при rate limit + анти-спам; не добавлять секреты в ответы API. | §2.2 |
| SEC-009 | A04 Insecure Design / A05 Security Misconfiguration | Опция C (Formspree, Getform): данные уходят на **инфраструктуру третьей стороны**; в спецификации не отражены требования к DPA, региону обработки (GDPR), срокам хранения. | MEDIUM | Выбрать провайдера с подходящей юрисдикцией; описать в политике конфиденциальности; не передавать лишние поля. | §2.3 |
| SEC-010 | A06 Vulnerable and Outdated Components | Стек (Next 15, GSAP, Lenis, зависимости почтового SDK) подвержен известным уязвимостям в зависимостях; в архитектуре указаны рекомендации по тестам, но **не** процесс обновления зависимостей. | LOW | Зафиксировать в пайплайне: lockfile, `npm audit`/Dependabot, контроль major-апгрейдов перед продом. | §1, §6 п.13 |
| SEC-011 | A03 Injection / A07 XSS | Контент из `src/data/` статичен; при соблюдении React-практик риск **Stored XSS** низок. Динамический `dangerouslySetInnerHTML` для JSON-LD должен использоваться только с сериализованными доверенными данными. | LOW | Запретить сырой HTML из непроверенных источников; для JSON-LD — безопасная сериализация из типизированных объектов. | §3.2 SEO |
| SEC-012 | A03 Injection | Отдельного backend и SQL в ТЗ нет; **SQL/NoSQL injection** не применимы к описанной модели. | INFO | Сохранять при добавлении БД или CMS: параметризованные запросы, принцип least privilege. | §1 |
| SEC-013 | A10 Server-Side Request Forgery (SSRF) | В спецификации нет серверных запросов по URL из пользовательского ввода. | INFO | При появлении превью ссылок, webhooks или прокси — не запрашивать произвольные URL без allowlist. | — |
| SEC-014 | A02 Cryptographic Failures | Деплой на Vercel предполагает **HTTPS**; для кастомного домена спецификация не требует явно принудительного HTTPS/HSTS (частично пересекается с SEC-006). | LOW | Включить HSTS на продакшене; не смешивать HTTP-ресурсы в контенте (mixed content). | §1, §6 |

## Итоговая оценка по OWASP Top 10 (уровень спецификации)

| OWASP | Актуальность для данного ТЗ |
|-------|-----------------------------|
| A01 Broken Access Control | Умеренная: CSRF и злоупотребление публичным POST. |
| A02 Cryptographic Failures | Низкая–средняя: секреты только server-side; публичный Mapbox. |
| A03 Injection | Средняя: email/header injection при опции B. |
| A04 Insecure Design | Средняя: отсутствие rate limit/CSRF в контракте; сторонние формы. |
| A05 Security Misconfiguration | Средняя: заголовки, ограничения токена Mapbox. |
| A06 Vulnerable Components | Низкая: процесс обновления не описан. |
| A07 Identification and Authentication Failures | Низкая: нет учётных записей пользователей сайта. |
| A08 Software and Data Integrity Failures | Низкая для статического контента. |
| A09 Security Logging and Monitoring Failures | Средняя: риск логирования PII при контактной форме. |
| A10 SSRF | Низкая при текущем scope. |

**Критических** (безусловный провал безопасности всего продукта только по тексту спецификации) не выявлено; **HIGH** относятся к обязательным дополнениям, если реализуется опция B без соответствующих контролей.
